# План доработки геосервиса

Дата составления: 2026-04-11

Основание:
- `docs/requirements-compliance-audit.md`
- `docs/geoservice-requirements.md`
- `docs/geoservice-prd-v1.md`

## Цель

Довести проект от текущего состояния технического MVP карты и CRUD до состояния, в котором он соответствует основным продуктовым и техническим требованиям.

## Принципы

- Сначала закрывать риски, которые мешают MVP-демо и приёмке.
- Не расширять доменную модель раньше, чем закрыты realtime, auth и ключевые пользовательские сценарии.
- Каждую доработку сопровождать тестами и обновлением документации.

## Спринт 1. Закрыть критичные MVP-gap'ы

Цель:
- Закрыть требования, без которых проект нельзя считать соответствующим MVP.

Задачи:
- Реализовать WebSocket realtime-подписку на слой.
- Добавить серверную публикацию событий `feature_created`, `feature_updated`, `feature_deleted`.
- Подключить frontend-подписку и обновление карты при входящих событиях.
- Реализовать production-ready login flow вместо dev-only сценария.
- Добавить базовые integration-тесты на auth и protected CRUD.

Результат спринта:
- второй клиент видит изменения почти в реальном времени;
- вход в систему возможен без dev-login;
- ключевой demo-сценарий совместного редактирования становится воспроизводимым.

## Спринт 2. Завершить пользовательский сценарий редактирования

Цель:
- Довести UI/UX до полного сценария работы `Viewer` и `Editor`.

Задачи:
- Добавить create-flow для новой feature на карте.
- Поддержать режим создания в edit-store и map interaction flow.
- Обобщить frontend-редактирование на все поддерживаемые типы геометрии, а не только на `Polygon`.
- Скрыть или дизейблить mutate-действия для `viewer`.
- Добавить integration/e2e тесты на create/update/delete.
- Обновить demo-checklist и сценарии тестирования.

Результат спринта:
- `Editor` может создать, изменить и удалить объект полностью через UI для всех поддерживаемых типов геометрии;
- `Viewer` получает корректный read-only интерфейс;
- CRUD сценарий закрыт не только API, но и пользовательским потоком.

## Спринт 3. История изменений и наблюдаемость

Цель:
- Сделать изменения объектов объяснимыми и трассируемыми.

Задачи:
- Добавить таблицу или модель аудита изменений feature.
- Логировать `create/update/delete` с пользователем, временем и типом изменения.
- Добавить endpoint просмотра истории feature.
- Добавить structured logging и request id.
- Написать тесты на историю изменений и audit flow.

Результат спринта:
- можно показать, кто и когда изменил объект;
- сервис становится лучше подготовлен к разбору ошибок и сопровождению.

## Спринт 4. Аналитика и продуктовая полировка

Цель:
- Добавить функции, которые повышают ценность сервиса сверх базового CRUD.

Задачи:
- Реализовать минимум 1-2 геоаналитические операции.
- Добавить UI или demo-flow для аналитики.
- Определить окончательное решение по сущности `Project`:
  - либо добавить её в модель и API;
  - либо официально исключить из MVP и поправить документацию.
- Усилить документацию по запуску, тестированию и demo-flow.

Результат спринта:
- требования по аналитике закрыты;
- доменная модель синхронизирована с реальным scope проекта;
- документация становится пригодной для защиты и передачи проекта.

## Технические задачи, которые идут сквозным потоком

- Поддерживать зелёный GitHub Actions pipeline.
- Расширять unit, integration и e2e покрытие после каждой ключевой доработки.
- Синхронизировать `docs/` после закрытия каждого крупного блока.
- Следить, чтобы новые feature не ломали performance карты и tile-cache.

## Рекомендуемый порядок реализации

1. WebSocket realtime
2. Production-ready auth
3. Integration-тесты auth + CRUD
4. Create-flow в UI и редактирование всех supported geometry types
5. Read-only UX для `viewer`
6. История изменений
7. Structured logging и request id
8. Геоаналитика
9. Решение по сущности `Project`

## Критерий завершения плана

План можно считать выполненным, когда:
- проект покрывает все обязательные требования из `geoservice-requirements.md`;
- demo-сценарий совместного редактирования проходит от начала до конца;
- CI остаётся зелёным;
- документация соответствует фактическому состоянию кода.
