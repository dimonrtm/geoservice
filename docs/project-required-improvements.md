# Наиболее необходимые доработки проекта

## Приоритет 1. Закрыть оставшиеся пробелы в качестве и тестировании

Базовый тестовый каркас уже есть: backend unit-тесты, frontend unit-тесты и проверки в GitHub Actions уже настроены. Теперь нужно добрать те уровни защиты, которых всё ещё не хватает.

Нужно сделать:

1. Добавить backend unit-тесты на конфликты версий и оставшиеся error-path сценарии сервисов.
2. Добавить integration-тесты CRUD для feature через API-слой.
3. Добавить e2e на сценарии создания и редактирования feature для всех поддерживаемых типов геометрии, а не только для полигонов.
4. Расширить frontend unit-тесты для map-composables и orchestration-логики загрузки/редактирования всех поддерживаемых `geometryType`.

Ожидаемый эффект:

- станет безопаснее продолжать рефакторинг;
- регрессии в map-flow и CRUD будут ловиться до merge;
- проект станет заметно надёжнее как GIS-MVP, а не только как набор модулей.

## Приоритет 2. Дожать безопасность production-конфигурации

Конфигурация уже вынесена в единый settings-модуль, а dev-auth скрыт за флагом. Но production-hardening ещё не завершён.

Нужно сделать:

1. Сделать обязательным безопасный `JWT_SECRET` вне dev-режима.
2. Убрать небезопасные значения по умолчанию для production-конфигурации.
3. Явно зафиксировать границу между dev-auth и рабочей auth-моделью в документации и настройках запуска.

Ожидаемый эффект:

- снизится риск случайно поднять небезопасную конфигурацию;
- эксплуатационное поведение backend станет предсказуемее;
- auth-слой будет проще сопровождать.

## Приоритет 3. Подготовить backend к росту данных

Стабильная сортировка, `meta.truncated`, `next_cursor` и tile-based loading уже реализованы. Теперь остались задачи, связанные именно с реальной производительностью базы и масштабированием.

Нужно сделать:

1. Проверить `EXPLAIN ANALYZE` для bbox-запросов на реальных объёмах данных.
2. Определить, нужен ли следующий шаг после текущего cursor/tile-based browse-слоя:
   - precomputed/simplified geometry;
   - или переход на vector tiles (`MVT`) для browse-сценария.
3. Зафиксировать лимиты и operational guidance для больших выборок.

Ожидаемый эффект:

- проект не будет деградировать неожиданно при росте объёма геоданных;
- станет понятен следующий performance-этап после текущего tile-cache;
- появится основа для масштабирования browse-сценария.

## Приоритет 4. Закрепить frontend после рефакторинга карты

Основной рефакторинг карты уже сделан, tile-cache и фоновая догрузка тоже внедрены. Но новую структуру ещё стоит окончательно стабилизировать.

Нужно сделать:

1. Расширить тесты на `useFeatureLoading` и сценарии фоновой догрузки tile-страниц.
2. Проверить, не нужны ли дополнительные UX-ограничения для background loading на больших viewport.
3. Обобщить edit-flow, overlay и map-interaction слой на все поддерживаемые типы геометрии, если где-то ещё осталась привязка только к полигонам.
4. При необходимости доразделить оставшиеся frontend-модули по той же модели ответственности, если ещё остались смешанные orchestration/helper участки.

Ожидаемый эффект:

- снизится риск регрессий в работе карты;
- дальнейшее развитие map-фич станет проще;
- поведение frontend под нагрузкой станет предсказуемее.

## Приоритет 5. Довести документацию до состояния "можно уверенно продолжать разработку"

Базовая документация и структура репозитория уже синхронизированы с кодом, но архитектурные решения всё ещё зафиксированы не полностью.

Нужно сделать:

1. Добавить ADR как минимум по темам:
   - стратегия browse/loading на карте;
   - auth/dev-auth границы;
   - контракт frontend/backend.
2. Описать ключевые пользовательские сценарии редактирования и просмотра данных.
3. Обновить roadmap в docs после закрытия оставшихся high-priority задач.

Ожидаемый эффект:

- проект станет понятнее для продолжения разработки;
- уменьшится разрыв между "как задумано" и "как реализовано";
- onboarding в кодовую базу станет проще.

## Рекомендуемая последовательность работ

Лучший порядок доработки сейчас такой:

1. Добрать integration/e2e и недостающие unit-тесты.
2. Закрыть production-hardening конфигурации и auth.
3. Проверить производительность bbox-запросов через `EXPLAIN ANALYZE`.
4. Закрепить новую frontend map-архитектуру дополнительными тестами, поддержкой всех supported geometry types и UX-ограничениями.
5. После этого зафиксировать всё в ADR и пользовательских сценариях.

## Что даст максимальную пользу в ближайшую неделю

Если нужен самый практичный short list, я бы делал именно это:

1. Добавить integration-тесты CRUD для feature API.
2. Добавить e2e на создание и редактирование feature всех поддерживаемых типов геометрии.
3. Сделать обязательным безопасный `JWT_SECRET` вне dev.
4. Прогнать и сохранить `EXPLAIN ANALYZE` для bbox-запросов.
5. Зафиксировать ADR по browse/loading контракту карты.

Это даст быстрый прирост надёжности и управляемости без нового большого архитектурного переписывания.
