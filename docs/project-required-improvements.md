# Наиболее необходимые доработки проекта

## Приоритет 1. Восстановить целостность основного пользовательского сценария

Это самый важный блок, потому что сейчас проект демонстрирует основной use case только частично.

Нужно сделать:

1. Починить обработку `GET feature by id`, чтобы не было `500` при отсутствии записи.

Ожидаемый эффект:

- чтение feature станет предсказуемым и консистентным;
- клиент перестанет получать внутреннюю ошибку вместо `404`;
- пользовательский путь станет воспроизводимым.

## Приоритет 2. Зафиксировать и выровнять API-контракты

Без этого backend и frontend будут и дальше расходиться.

Нужно сделать:

1. Описать контракт ответов для `layers`, `feature collection`, `feature`, `patch conflict`, `delete response`.
2. Выбрать единый формат ответов: либо "голые DTO", либо обёртки вида `{ data: ... }`.
3. Удалить неиспользуемые схемы и временные переходные конструкции.
4. По возможности вынести контракт в общий слой или генерируемый клиент.

Ожидаемый эффект:

- меньше ручных парсеров и рассинхронизации;
- проще писать интеграционные тесты;
- проще менять API без поломки UI.

## Приоритет 3. Ужесточить границы dev-режима и безопасности

Сейчас dev-механики слишком легко смешиваются с рабочим приложением.

Нужно сделать:

1. Сделать обязательным задание `JWT_SECRET` вне dev-режима.
2. Убрать небезопасные значения по умолчанию.
3. Скрыть `CheckAuthComponent` за dev flag.
4. Явно разделить dev-auth и рабочую auth-модель.
5. Перенести конфигурацию в единый settings-модуль.

Ожидаемый эффект:

- меньше риск случайно развернуть небезопасный вариант;
- архитектура auth станет чище и понятнее.

## Приоритет 4. Добавить реальное тестирование

Сейчас проект почти не защищён от регрессий.

Нужно сделать:

1. Добавить backend unit-тесты для:
   - `parse_bbox`;
   - `FeatureService`;
   - конфликтов версий;
   - ошибок отсутствия слоя и feature.
2. Добавить frontend unit-тесты для:
   - геометрических преобразований;
   - `edit` store;
   - runtime-validators.
3. Добавить интеграционные тесты CRUD для feature.
4. Добавить e2e для сценария редактирования полигона.

Ожидаемый эффект:

- станет безопаснее рефакторить;
- дефекты уровня текущих уже будут ловиться до merge.

## Приоритет 5. Добавить frontend-проверки в CI

Сейчас именно этот разрыв уже привёл к поломанной сборке в основной ветке.

Нужно сделать:

1. Добавить `npm run lint`.
2. Добавить `npm run typecheck`.
3. Добавить `npm run build`.
4. Сделать их обязательными для merge.

Ожидаемый эффект:

- broken frontend перестанет попадать в основную ветку.

## Приоритет 6. Закрепить frontend после рефакторинга

Основной рефакторинг карты уже сделан и шаблонный мусор уже убран, но новую структуру ещё нужно закрепить тестами и правилами CI.

Нужно сделать:

1. Добавить unit-тесты на вынесенную геометрию и map-composables.
2. Привести оставшиеся frontend-модули к той же новой структуре ответственности, если они ещё не синхронизированы с composables.

Ожидаемый эффект:

- снизится риск регрессий;
- разработка новых map-features ускорится.

## Приоритет 7. Подготовить backend к росту данных

Сейчас backend подходит для малого объёма данных, но не для масштабирования.

Нужно сделать:

1. Добавить `GIST`-индексы на все `geom`-колонки.
2. Проверить `EXPLAIN ANALYZE` для bbox-запросов.
3. Определить стратегию выборки:
   - лимиты;
   - пагинация;
   - tile-based loading.

Ожидаемый эффект:

- проект перестанет резко деградировать при росте объёма геоданных.

## Приоритет 8. Довести документацию до состояния "можно уверенно продолжать разработку"

Базовая документация и структура уже синхронизированы с кодом, но архитектурные решения и пользовательские сценарии всё ещё не зафиксированы.

Нужно сделать:

1. Заполнить `docs/adr` как минимум тремя решениями:
   - хранение геометрий;
   - стратегия auth;
   - контракт frontend/backend.
2. Заполнить `docs/user-stories` основными пользовательскими сценариями.

Ожидаемый эффект:

- проект станет понятнее для продолжения разработки;
- уменьшится разрыв между "как задумано" и "как реализовано".

## Рекомендуемая последовательность работ

Лучший порядок доработки такой:

1. Починить обработку `GET feature by id`.
2. Синхронизировать API-контракты.
3. Добавить тесты и CI-проверки фронтенда.
4. Ограничить dev-auth и привести в порядок конфигурацию.
5. После этого заняться оптимизацией запросов и индексами.
6. Затем закрепить frontend и архитектурную документацию.

## Что даст максимальную пользу в ближайшую неделю

Если нужен самый практичный short list, я бы делал именно это:

1. Починить `GET feature by id`, чтобы он возвращал `404`, а не `500`.
2. Добавить unit-тесты на геометрию и `FeatureService`.
3. Добавить frontend build/lint/typecheck в CI.
4. Добавить spatial indexes.
5. Начать писать ADR по геометриям и API-контрактам.

Это даст быстрый прирост надёжности без большого архитектурного переписывания.
