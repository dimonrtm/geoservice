# Анализ проекта GeoService

## Что это за проект

`GeoService` — это экспериментальный MVP-сервис для работы с геоданными. По `README.md` проект задуман как приложение для:

- загрузки GeoJSON;
- хранения геоданных в PostGIS;
- отображения объектов на карте;
- базового совместного редактирования геометрий.

По фактическому состоянию кода это монорепозиторий с двумя основными приложениями:

- `apps/backend/app` — backend на `FastAPI + SQLAlchemy + PostGIS + Alembic`;
- `apps/frontend` — frontend на `Vue 3 + TypeScript + Pinia + MapLibre`.

Инфраструктура поднимается через Docker Compose из `infra/docker-compose.yml`. База данных — `postgis/postgis:16-3.4`. На бэкенде реализованы операции чтения слоёв и CRUD для feature-объектов, на фронтенде есть карта, выбор слоя, загрузка объектов по `bbox` и зачатки редактирования полигонов.

## Реальное назначение и стадия зрелости

По коду это не production-ready GIS-платформа, а именно учебно-экспериментальный MVP:

- есть базовый каркас доменных сущностей и API;
- есть dev-only аутентификация без полноценной модели безопасности;
- часть возможностей недоделана или разошлась между frontend и backend;
- документация и пакет `packages/shared/schemas` пока пустые;
- тестовое покрытие практически отсутствует.

Иными словами, проект уже демонстрирует основную бизнес-идею, но пока находится на стадии "прототип, который нужно довести до консистентного MVP".

## Текущая архитектура

### Backend

Слои backend-а в текущем виде:

- `api/` — FastAPI роуты, зависимости и dev-auth;
- `services/` — прикладная логика работы со слоями, фичами и dev-пользователями;
- `repositories/` — доступ к БД через SQLAlchemy;
- `models/` — SQLAlchemy-модели;
- `domain/` — доменные исключения, `bbox`, реестр моделей по типам геометрии;
- `alembic/` — миграции схемы БД.

Главная особенность модели данных: для каждого типа геометрии используется отдельная таблица:

- `feature_points`
- `feature_lines`
- `feature_polygons`
- `feature_multipoints`
- `feature_multilines`
- `feature_multipolygons`

Сущность `layers` хранит метаданные слоя и имя физической таблицы хранения (`storage_table`). Дальше `feature_registry.py` по `storage_table` выбирает SQLAlchemy-модель.

Это рабочая схема для MVP, но она уже создаёт усложнение: часть логики становится динамической, а контракт между слоем и физической таблицей не гарантируется типами или внешними ключами.

### Frontend

Frontend состоит из нескольких зон ответственности:

- `src/api/` — HTTP-клиент и вызовы backend API;
- `src/stores/` — auth- и edit-store на Pinia;
- `src/components/` — UI-компоненты, включая карту;
- `src/map/` — работа с MapLibre, слоями и геометрическим редактированием;
- `src/parsing/` — runtime-проверки данных и ошибок.

Ключевой сценарий строится вокруг `MapView.vue`: там одновременно находятся:

- инициализация карты;
- загрузка слоёв;
- загрузка feature-объектов по `bbox`;
- реакция на перемещение карты;
- выбор объекта;
- управление состоянием редактирования;
- обработка drag/drop вершин;
- UI-сообщения.

То есть основной компонент карты уже стал "god component".

## Что уже работает концептуально

На уровне концепции проект уже покрывает полезный путь пользователя:

1. пользователь получает токен через dev-login;
2. выбирает слой;
3. карта запрашивает объекты по `bbox`;
4. объекты рисуются на MapLibre;
5. для полигонов запускается режим редактирования;
6. backend поддерживает optimistic locking по `version`.

Это хороший фундамент для GIS-MVP: есть карта, есть слои, есть spatial query, есть модель конкурентного редактирования.

## Что не соответствует заявленной структуре

Есть заметный разрыв между описанием проекта и фактическим состоянием репозитория:

- в `README.md` перечислены директории `core/`, `ui/`, `state/`, `maplayer/`, `apiclient/`, `styles/`, но в реальном коде структура другая;
- `docs/adr`, `docs/user-stories` и `packages/shared/schemas` существуют только как пустые директории;
- `apps/frontend/README.md` всё ещё содержит шаблонный текст от Vite/Vue;
- backend и frontend развиваются не из общего контракта, а фактически независимо.

Это показывает, что архитектурный замысел был шире текущей реализации, но репозиторий ещё не приведён к целостному виду.

## Итоговая оценка проекта

Проект — это геосервис для визуализации и редактирования геообъектов, собранный как монорепозиторий с современным стеком и понятной MVP-целью. Сильная сторона проекта — правильно выбранная технологическая база и уже работающая доменная идея. Слабая сторона — несогласованность между слоями, высокая концентрация логики в нескольких файлах, пустая документация, минимальное тестирование и недоведённость ключевых сценариев редактирования.

На текущем этапе проект лучше всего описывается как:

"экспериментальный GIS-MVP с хорошим технологическим фундаментом, но с незавершённой интеграцией между frontend, backend и контрактами данных".
