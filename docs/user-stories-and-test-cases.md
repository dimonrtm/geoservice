## 1. User Stories

### US-1. Вход в систему

Как пользователь,
я хочу войти в систему и получить токен,
чтобы использовать защищённые функции геосервиса.

### US-2. Просмотр слоя на карте

Как пользователь,
я хочу открыть слой и видеть объекты на карте в пределах текущего viewport,
чтобы работать с нужной областью данных.

### US-3. Создание объекта

Как `Editor`,
я хочу создать новый объект на карте,
чтобы добавить новые геоданные в слой.

### US-4. Редактирование объекта

Как `Editor`,
я хочу изменить геометрию или атрибуты объекта,
чтобы актуализировать информацию.

### US-5. Удаление объекта

Как `Editor`,
я хочу удалить объект,
чтобы убрать ошибочные или устаревшие данные.

### US-6. Realtime-обновления

Как пользователь,
я хочу видеть изменения других пользователей почти сразу,
чтобы работать с актуальным состоянием слоя.

### US-7. Разрешение конфликта версий

Как `Editor`,
я хочу получить понятную ошибку конфликта и перезагрузить актуальный объект,
чтобы не потерять чужие изменения.

### US-8. Просмотр истории изменений

Как пользователь продукта,
я хочу видеть минимальную историю изменений объекта,
чтобы понимать, кто и когда его менял.

## 2. Основные acceptance-сценарии в формате Given/When/Then

### AC-1. Успешная авторизация

Given:
- пользователь существует в системе;
- пользователь вводит корректные учётные данные.

When:
- пользователь отправляет запрос на вход.

Then:
- сервер возвращает успешный ответ;
- клиент получает JWT-токен;
- токен используется в последующих защищённых запросах.

### AC-2. Запрет доступа без токена

Given:
- пользователь не авторизован.

When:
- пользователь обращается к защищённому API.

Then:
- сервер возвращает `401 Unauthorized`.

### AC-3. Просмотр слоя на карте

Given:
- пользователь авторизован;
- в системе существует слой и объекты в пределах `bbox`.

When:
- пользователь открывает карту;
- клиент отправляет запрос на загрузку объектов по `bbox`.

Then:
- сервер возвращает GeoJSON-объекты слоя;
- объекты отображаются на карте.

### AC-4. Создание объекта

Given:
- пользователь авторизован как `Editor`;
- открыт слой с поддерживаемым `geometryType`.

When:
- пользователь рисует новый объект на карте;
- клиент отправляет `POST` с валидной геометрией.

Then:
- сервер сохраняет объект;
- сервер возвращает созданную feature с версией;
- объект появляется на карте.

### AC-5. Редактирование объекта

Given:
- пользователь авторизован как `Editor`;
- объект существует;
- у клиента есть актуальная версия объекта.

When:
- пользователь изменяет геометрию или свойства;
- клиент отправляет `PATCH/PUT` с корректной версией.

Then:
- сервер сохраняет изменения;
- версия объекта увеличивается;
- клиент отображает обновлённый объект.

### AC-6. Удаление объекта

Given:
- пользователь авторизован как `Editor`;
- объект существует;
- клиент знает актуальную версию объекта.

When:
- пользователь удаляет объект;
- клиент отправляет `DELETE` с `version`.

Then:
- сервер удаляет объект;
- сервер возвращает успешный ответ удаления;
- объект исчезает с карты.

### AC-7. Realtime-обновление на втором клиенте

Given:
- пользователь A и пользователь B открыли один и тот же слой;
- оба подписаны на WebSocket-канал слоя.

When:
- пользователь A создаёт, изменяет или удаляет объект.

Then:
- сервер рассылает соответствующее событие;
- пользователь B видит изменение на карте не позднее чем через `1–2 секунды`.

### AC-8. Конфликт версий

Given:
- у feature текущая версия `7`;
- пользователь A и пользователь B одновременно открыли один и тот же объект.

When:
- пользователь A сохраняет объект и получает новую версию `8`;
- пользователь B пытается сохранить старую версию `7`.

Then:
- сервер возвращает `409 VERSION_MISMATCH`;
- клиент B показывает сообщение о конфликте;
- клиент B сбрасывает локальные изменения;
- клиент B выполняет `GET` актуальной feature;
- после загрузки версии `8` редактирование снова становится доступно.

### AC-9. Ограничение прав Viewer

Given:
- пользователь авторизован как `Viewer`.

When:
- пользователь пытается вызвать `POST`, `PATCH` или `DELETE`.

Then:
- сервер возвращает `403 Forbidden`.

### AC-10. Валидация геометрии и bbox

Given:
- клиент отправляет геометрию или `bbox`.

When:
- геометрия не соответствует типу слоя;
- или координаты вне допустимого диапазона;
- или `bbox` невалиден.

Then:
- сервер возвращает `422 Unprocessable Entity`;
- ответ содержит типизированную ошибку.

## 3. Ключевые ручные test cases

### TC-1. Login и доступ к защищённым эндпойнтам

Предусловия:
- backend и frontend запущены;
- есть тестовый пользователь.

Шаги:
1. Выполнить вход.
2. Открыть карту.
3. Проверить успешный запрос к защищённому API.

Ожидаемый результат:
- токен получен;
- защищённые запросы работают;
- без токена те же запросы дают `401`.

### TC-2. Viewer не может редактировать

Предусловия:
- пользователь вошёл как `Viewer`.

Шаги:
1. Попробовать создать объект.
2. Попробовать изменить объект.
3. Попробовать удалить объект.

Ожидаемый результат:
- все write-операции отклоняются с `403`.

### TC-3. Editor создаёт объект

Предусловия:
- пользователь вошёл как `Editor`;
- выбран слой нужного типа.

Шаги:
1. Нарисовать объект на карте.
2. Сохранить его.

Ожидаемый результат:
- объект появляется на карте;
- backend возвращает новую feature с версией;
- второй клиент получает событие создания.

### TC-4. Editor изменяет объект

Предусловия:
- существует feature;
- пользователь вошёл как `Editor`.

Шаги:
1. Выбрать объект.
2. Изменить геометрию или атрибуты.
3. Сохранить изменения.

Ожидаемый результат:
- версия увеличивается;
- карта отображает новую геометрию;
- второй клиент видит изменение по WebSocket.

### TC-5. Editor удаляет объект

Предусловия:
- существует feature;
- пользователь вошёл как `Editor`.

Шаги:
1. Выбрать объект.
2. Выполнить удаление.

Ожидаемый результат:
- объект удалён из БД;
- объект исчезает с карты;
- второй клиент видит удаление.

### TC-6. Конфликт версий между двумя вкладками

Предусловия:
- открыты вкладки A и B;
- в обеих загружена одна и та же feature с одной и той же версией.

Шаги:
1. Во вкладке A изменить feature и сохранить.
2. Во вкладке B, не перезагружая объект, попытаться сохранить старую версию.

Ожидаемый результат:
- вкладка A получает успех;
- вкладка B получает `409`;
- вкладка B загружает актуальную версию;
- локальные устаревшие изменения вкладки B сбрасываются.

### TC-7. Валидация неверной геометрии

Предусловия:
- пользователь вошёл как `Editor`.

Шаги:
1. Отправить feature с неподдерживаемым `geometry.type`;
или
2. Отправить feature с координатами вне диапазона.

Ожидаемый результат:
- сервер возвращает `422`;
- ошибка содержит ожидаемый `type`.

### TC-8. Валидация bbox

Предусловия:
- пользователь авторизован.

Шаги:
1. Отправить запрос `GET features` с невалидным `bbox`.

Ожидаемый результат:
- сервер возвращает `422 INVALID_BBOX`.

### TC-9. Reconnect WebSocket

Предусловия:
- клиент подписан на слой.

Шаги:
1. Искусственно разорвать соединение.
2. Восстановить его.
3. Изменить объект с другого клиента.

Ожидаемый результат:
- клиент переподписывается;
- новые изменения снова доходят;
- локальное состояние остаётся консистентным.

## 4. Минимальный набор автоматизации

### Backend integration tests

- успешный login;
- `401` без токена;
- `403` для `Viewer` на write-endpoints;
- CRUD feature;
- `409 VERSION_MISMATCH`;
- `422 INVALID_BBOX`;
- `422 INVALID_COORDINATES`;
- `422 UNSUPPORTED_GEOMETRY`.

### Frontend tests

- smoke тест открытия карты;
- создание объекта;
- редактирование объекта;
- удаление объекта;
- обработка `409`;
- применение WebSocket patch-событий.

### E2E demo tests

- два клиента открывают один слой;
- клиент A изменяет feature;
- клиент B видит изменение по WebSocket;
- клиент B получает `409` при сохранении старой версии;
- клиент B загружает актуальную feature.

## 5. Demo-сценарий для Sprint Review

### Сценарий 1. Базовый happy path

1. Пользователь входит как `Editor`.
2. Открывает слой на карте.
3. Создаёт объект.
4. Изменяет объект.
5. Удаляет объект.

Ожидаемый результат:
- все три операции проходят успешно;
- карта остаётся консистентной;
- backend возвращает корректные версии объектов.

### Сценарий 2. Совместное редактирование

1. Открыть две вкладки A и B.
2. В обеих открыть один и тот же слой.
3. Во вкладке A изменить объект.
4. Проверить, что вкладка B видит обновление почти сразу.

Ожидаемый результат:
- изменения синхронизируются по WebSocket не позднее чем за `1–2 секунды`.

### Сценарий 3. Конфликт

1. Открыть две вкладки A и B.
2. Загрузить одну и ту же feature.
3. Во вкладке A сохранить изменения.
4. Во вкладке B попытаться сохранить устаревшую версию.

Ожидаемый результат:
- вкладка B получает `409`;
- приложение показывает уведомление;
- выполняется перезагрузка актуальной feature;
- пользователь может продолжить работу уже с новой версией.

## 6. Non-goals для тестирования MVP

В рамках MVP не требуется покрывать полноценными тест-кейсами:

- CRDT/OT;
- сложные lock-сценарии;
- offline-sync;
- расширенные права доступа на уровне отдельных атрибутов;
- сложные топологические геопроверки.
