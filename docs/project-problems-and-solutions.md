# Проблемы проекта и варианты решений

## Ключевой вывод

Главная проблема проекта не в выборе технологий, а в несогласованности реализации. Архитектурно backend и frontend движутся в правильную сторону, но между ними уже накопились расхождения в API-контрактах, ответственности модулей и уровне зрелости. Из-за этого часть системы выглядит готовой, но фактически сценарии ломаются на интеграции.

## 1. Архитектурные проблемы

### 1.1. Архитектура frontend слишком концентрирована в одном компоненте

Признаки:

- `apps/frontend/src/components/MapView.vue` содержит почти всю логику карты, загрузки данных и редактирования;
- `apps/frontend/src/map/maplibrelayers.ts` одновременно отвечает и за визуальные слои, и за редактирование геометрии, и за геометрические вычисления.

Последствия:

- код трудно тестировать;
- любое изменение карты затрагивает сразу несколько сценариев;
- высокий риск регрессий;
- вход в проект для нового разработчика сложнее, чем должен быть.

Что делать:

- разделить `MapView.vue` на composables и специализированные модули:
  - `useMapInstance`
  - `useLayerSelection`
  - `useFeatureLoading`
  - `usePolygonEditing`
- вынести математику геометрий в отдельный чистый модуль без зависимостей от MapLibre;
- оставить в компоненте только orchestration и шаблон.

### 1.2. Модель данных по геометриям усложняет систему быстрее, чем даёт выгоду

Признаки:

- для каждого типа геометрии заведена отдельная таблица и отдельная модель;
- выбор модели происходит динамически через `feature_registry.py`;
- слой хранит строку `storage_table`, по которой находится физическая таблица.

Последствия:

- усложняются репозитории, миграции и тесты;
- появляются дополнительные точки рассинхронизации между `layers.storage_table`, миграциями и registry;
- расширение типов геометрий увеличивает стоимость сопровождения.

Что делать:

- для MVP рассмотреть единую таблицу `features` с колонками `layer_id`, `geom`, `properties`, `version`;
- если разделение по типам сохраняется, добавить строгие проверки консистентности:
  - системные тесты registry;
  - ограничения и seed-проверки;
  - явный слой domain mapping вместо строковых значений.

### 1.3. Конфигурация и безопасность размазаны по коду

Признаки:

- `DEV_MODE=True` по умолчанию;
- `JWT_SECRET` имеет небезопасное значение по умолчанию;
- dev-login встроен прямо в runtime API;
- dev auth panel всегда рендерится в `App.vue`;
- токен хранится в `localStorage`.

Последствия:

- dev-механизмы легко попадут в non-dev окружение;
- приложение слабо защищено даже для внутреннего стенда;
- логика безопасности не отделена от удобства разработки.

Что делать:

- вынести конфигурацию в единый settings-модуль;
- сделать небезопасные значения фатальной ошибкой старта;
- спрятать dev-login и dev UI за явным feature flag;
- для production-пути уйти от `localStorage` хотя бы к более безопасной модели токенов.

### 1.4. Документация и структура репозитория не синхронизированы с кодом

Признаки:

- `README.md` описывает структуру, которой в репозитории нет;
- `docs/` и `packages/shared/schemas` пустые;
- `apps/frontend/README.md` остался шаблонным.

Последствия:

- архитектурная картина в голове команды и в коде расходится;
- onboarding замедляется;
- решения принимаются без зафиксированных ADR и user stories.

Что делать:

- привести `README.md` к реальному состоянию;
- завести хотя бы минимальные ADR по хранению геометрий, auth и sync frontend/backend контрактов;
- либо наполнить `packages/shared/schemas`, либо удалить декларативный пакет до реальной необходимости.

## 2. Code Smells и дефекты реализации

### 2.1. Фронтенд не собирается

Факт:

- `npm run build` падает на `apps/frontend/src/components/MapView.vue` из-за незавершённой функции `onOutlineInsert`.

Почему это важно:

- это не "потенциальный smell", а уже воспроизводимый блокер поставки.

Что делать:

- либо закончить вставку вершины;
- либо временно удалить незавершённый код и закрыть фичу флагом;
- добавить обязательную frontend-сборку в CI.

### 2.2. Незавершённая и частично мёртвая функциональность загрузки GeoJSON

Признаки:

- `FileLoader.vue` вызывает `POST http://localhost:8000/api/geojson/import`;
- такого endpoint на backend нет;
- компонент не интегрирован в основной пользовательский путь.

Последствия:

- код создаёт ложное ощущение поддерживаемой фичи;
- разработчик или пользователь тратит время на несуществующий сценарий.

Что делать:

- либо реализовать endpoint и полноценно встроить загрузку;
- либо удалить компонент до появления серверной части;
- не держать в main-ветке UI без backend-поддержки.

### 2.3. Глобальная обработка `ValueError` как HTTP 422 скрывает реальные ошибки

Признаки:

- в `apps/backend/app/main.py` любой `ValueError` переводится в пользовательскую ошибку `422`.

Последствия:

- программистские ошибки могут маскироваться под validation error;
- диагностика продакшн-инцидентов усложняется;
- журналы теряют диагностическую ценность.

Что делать:

- использовать отдельные прикладные исключения для бизнес-валидации;
- не ловить глобально `ValueError`;
- неожиданные ошибки оставлять как `500` с логированием.

### 2.4. Получение feature по id не обрабатывает отсутствие записи

Признаки:

- `FeatureService.get_feature()` берёт `row = await repository.get_feature(...)`;
- дальше код сразу обращается к `row.id`, `row.version`, `row.properties`;
- при `row is None` вернётся не `404`, а внутренняя ошибка.

Последствия:

- контракт API нарушается;
- клиент получает неконсистентное поведение между GET, PATCH и DELETE.

Что делать:

- добавить явную проверку `row is None`;
- выбрасывать `FeatureNotFoundException`.

### 2.5. В edit-логике есть скрытые логические ошибки

Признаки:

- `removePolygonVertex()` и `insertVertexOnNearestSegment()` изменяют `uncloseRing`, но не записывают результат обратно в `nextCoords[ringIndex]`;
- проверки вида `if (!u_j[0] || !u_j[1])` считают координату `0` невалидной, хотя это валидное значение;
- `onOutlineInsert()` недописан.

Последствия:

- редактирование работает непредсказуемо;
- геометрия может "не меняться", хотя пользователь ожидал изменение;
- объекты у нулевого меридиана/экватора могут обрабатываться ошибочно.

Что делать:

- покрыть геометрические операции unit-тестами;
- переписать функции как чистые детерминированные преобразования;
- убрать falsy-проверки чисел и заменить их на `Number.isFinite`.

### 2.6. Слишком много дублирования и слабой типизации

Признаки:

- backend хранит геометрию как `dict[str, Any]`;
- frontend делает большое количество runtime-проверок вручную;
- есть дублирование env-парсеров и повторяющихся CRUD-паттернов;
- набор моделей feature почти полностью повторяется.

Последствия:

- система менее надёжна, чем кажется по аннотациям;
- стоимость изменений растёт;
- шанс на расхождение контрактов повышается.

Что делать:

- ввести типизированные GeoJSON-схемы;
- централизовать работу с env;
- унифицировать общую feature-модель;
- использовать общий contract layer для frontend/backend.

### 2.7. В репозитории есть артефакты незавершённости

Примеры:

- `api/__init__` без расширения;
- `domain/___init__.py` с опечаткой;
- комментарий "поправь путь импорта под свою структуру" в `secure_router.py`;
- шаблонный `HelloWorld.vue`;
- `print` и `console.log` в runtime-коде.

Последствия:

- снижается доверие к целостности проекта;
- становится сложнее отличать черновик от рабочего кода.

Что делать:

- провести cleanup-итерацию по репозиторию;
- удалить шаблонные и временные файлы;
- включить линтерные правила против debug-логов в production-коде.

## 3. Проблемы производительности

### 3.1. Нет пространственных индексов на геометриях

Признаки:

- в миграциях создаются таблицы с `geom`, но нет `GIST/SP-GiST` индексов.

Последствия:

- `bbox`-запросы будут быстро деградировать при росте объёма данных;
- `ST_Intersects` начнёт сканировать слишком много строк.

Что делать:

- добавить `GIST`-индексы для `geom` во всех feature-таблицах;
- проверить планы выполнения через `EXPLAIN ANALYZE`.

### 3.2. Клиент запрашивает целые GeoJSON-коллекции при каждом смещении карты

Признаки:

- после `moveend` фронтенд перезапрашивает объекты по `bbox`;
- данные целиком переводятся в GeoJSON и передаются в клиент;
- затем весь source обновляется через `source.setData(...)`.

Последствия:

- лишний сетевой трафик;
- лишняя сериализация на backend;
- лишняя переработка данных в браузере;
- при росте набора данных карта начнёт "тяжелеть".

Что делать:

- ввести кэширование по `bbox`/zoom/layer;
- перейти на тайлы или векторные тайлы для больших данных;
- ограничить сценарий редактирования выборкой активных объектов, а не полной перезагрузкой.

### 3.3. `ST_AsGeoJSON` вызывается на каждую запись и каждый запрос

Признаки:

- backend на каждый запрос сериализует геометрию в JSON строку, затем сервис снова парсит JSON.

Последствия:

- лишняя CPU-нагрузка на backend;
- лишняя сериализация/десериализация;
- стоимость запроса выше необходимой.

Что делать:

- оценить, можно ли сократить объём геометрии или отдавать только нужные поля;
- для больших сценариев перейти на другой формат доставки данных;
- не повторять лишние преобразования в service-слое.

### 3.4. Нет серверной пагинации и стабильного порядка выдачи

Признаки:

- есть только `limit`, но нет `offset`, курсора, сортировки и стабильного порядка.

Последствия:

- клиент не контролирует полноту выборки;
- результат может быть нестабильным;
- масштабирование поиска и листинга ограничено.

Что делать:

- определить контракт выборки: сортировка, курсор или tile-based pagination;
- разделить сценарии просмотра и сценарии редактирования.

## 4. Проблемы качества и тестирования

### 4.1. Тестов почти нет

Факт:

- в backend есть только `test_sanity.py` с `assert 1 + 1 == 2`;
- frontend-тестов нет.

Последствия:

- регрессии ловятся только вручную;
- refactoring дорог и опасен;
- функциональные ошибки уже попали в основную ветку.

Что делать:

- минимум: unit-тесты для `bbox`, feature service и геометрических операций фронтенда;
- затем: интеграционные тесты API с тестовой БД;
- затем: e2e на сценарии "логин -> выбрать слой -> выбрать объект -> изменить -> сохранить".

### 4.2. CI проверяет backend, но почти не защищает frontend

Факт:

- `.gitlab-ci.yml` покрывает backend format/lint/test/build и smoke test;
- frontend build/lint/typecheck в CI отсутствуют.

Последствия:

- фронтенд уже сломан, но пайплайн этого не страхует.

Что делать:

- добавить в CI как минимум:
  - `npm run lint`
  - `npm run typecheck`
  - `npm run build`

## 5. Сводный приоритет проблем

Самые критичные проблемы прямо сейчас:

1. Сломанная frontend-сборка и незавершённый сценарий редактирования.
2. Отсутствие тестов и frontend-проверок в CI.
3. Небезопасный dev-auth по умолчанию.
4. Отсутствие spatial indexes и стратегии масштабирования выборок.

Если исправить только эти четыре пункта, проект уже станет заметно устойчивее, понятнее и ближе к реальному MVP.
