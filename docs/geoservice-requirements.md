# Требования к геосервису

## 1. Назначение системы

Геосервис должен обеспечивать MVP совместного редактирования геоданных в браузере на базе `Vue + MapLibre` и `FastAPI + PostGIS` с базовой авторизацией, realtime-обновлениями и защитой от конфликтов редактирования.

## 2. Scope MVP

В рамках MVP система должна поддерживать:

- просмотр слоёв и объектов на 2D-карте;
- создание, изменение и удаление геообъектов;
- одновременную работу минимум двух пользователей;
- доставку изменений почти в реальном времени;
- обработку конфликтов редактирования через optimistic concurrency;
- базовую CI/CD-готовность и воспроизводимый запуск.

В рамках MVP система не обязана поддерживать:

- `CRDT/OT`;
- офлайн-режим и последующую синхронизацию;
- сложные ACL на уровне объекта или поля;
- полноценные блокировки редактирования;
- сложную топологическую валидацию;
- multi-object транзакции;
- расширенный audit trail сверх минимально необходимого.

## 3. Пользователи и роли

Система должна поддерживать минимум две роли:

- `Viewer` — может только читать данные;
- `Editor` — может читать и изменять данные.

## 4. Основные сущности

Система должна оперировать следующими сущностями:

- `User`;
- `Project`;
- `Layer`;
- `Feature`;
- `FeatureEdit` — опционально, для истории изменений.

## 5. Требования к данным

- Формат геоданных: `GeoJSON`.
- Пространственная система координат: `SRID=4326`.
- Порядок координат: `[lon, lat]`.
- Формат `bbox`: `minLon,minLat,maxLon,maxLat`.
- Версия объекта должна храниться как верхнеуровневое поле `version` или `revision`.
- У слоя должен быть фиксированный `geometryType`.
- `Feature.geometry.type` обязан совпадать с `Layer.geometryType`.
- Допустимые типы геометрии:
  - `Point`
  - `LineString`
  - `Polygon`
  - `MultiPoint`
  - `MultiLineString`
  - `MultiPolygon`
- Координаты должны валидироваться:
  - `lon ∈ [-180, 180]`
  - `lat ∈ [-90, 90]`
- `bbox` должен валидироваться:
  - `minLon < maxLon`
  - `minLat < maxLat`
  - все значения в допустимом диапазоне

## 6. Функциональные требования

### 6.1. Авторизация и доступ

- Пользователь должен иметь возможность войти и получить JWT-токен.
- Все защищённые эндпойнты должны требовать валидный токен.
- Неавторизованный пользователь должен получать `401 Unauthorized`.
- Пользователь с ролью `Viewer` при попытке выполнить `POST`, `PATCH`, `PUT` или `DELETE` должен получать `403 Forbidden`.

### 6.2. Работа со слоями

- Сервис должен отдавать список доступных слоёв.
- Пользователь должен иметь возможность запрашивать объекты слоя по `bbox`.
- Объекты должны отдаваться в формате GeoJSON.
- Для картографического сценария сервис должен поддерживать выборку по viewport и ограничение объёма ответа.

### 6.3. Работа с объектами

- `Editor` должен иметь возможность:
  - создать объект;
  - изменить объект;
  - удалить объект.
- После успешного изменения сервис должен возвращать актуальную версию объекта.
- Для удаления клиент должен передавать версию объекта.
- При изменении объекта должна выполняться валидация геометрии и координат.

### 6.4. Отображение на карте

- Пользователь должен видеть слой на 2D-карте.
- Клиент должен загружать объекты по `bbox` при перемещении карты.
- Запросы по карте должны поддерживать debounce и отмену устаревших запросов.
- Клиент должен уметь обновлять данные слоя без полной перезагрузки приложения.

### 6.5. Совместное редактирование

- Второй пользователь должен видеть изменения первого в реальном времени.
- Допустимая задержка доставки изменений: не более `1–2 секунд`.
- Realtime-механизм должен быть реализован через WebSocket-подписку на слой.

### 6.6. Realtime-события

- Клиент должен иметь возможность подписаться на `layer_id`.
- Сервер должен рассылать события:
  - `feature_created`
  - `feature_updated`
  - `feature_deleted`
- Повторная доставка события не должна ломать состояние клиента.
- После потери соединения клиент должен иметь возможность переподписаться и восстановить актуальное состояние.

### 6.7. Конфликты редактирования

- Изменение объекта должно выполняться только если `request.version == currentVersion`.
- При конфликте сервис должен возвращать `409 Conflict`.
- Ответ `409` должен содержать:
  - тип ошибки;
  - `featureId`;
  - `requestVersion`;
  - `currentVersion`;
  - сообщение для пользователя.
- После `409` клиент должен:
  - распознать конфликт;
  - показать уведомление;
  - сбросить локальные изменения;
  - перезагрузить актуальную feature;
  - разблокировать повторное редактирование только после успешного `GET`.

### 6.8. История изменений

- Сервис должен уметь логировать изменения объектов: кто, когда и что изменил.
- Должен существовать минимум один эндпойнт для просмотра истории объекта.

### 6.9. Мини-аналитика

Система должна поддерживать минимум 1–2 пространственные аналитические операции, например:

- количество объектов в `bbox`;
- площадь выбранного полигона;
- поиск объектов в радиусе от точки.

## 7. API-требования

Минимально необходимы следующие эндпойнты:

- `GET /api/v1/layers`
- `GET /api/v1/layers/{layerId}/features?bbox=...&limit=...`
- `GET /api/v1/layers/{layerId}/features/{featureId}`
- `POST /api/v1/layers/{layerId}/features`
- `PATCH /api/v1/layers/{layerId}/features/{featureId}` или `PUT /api/v1/layers/{layerId}/features/{featureId}`
- `DELETE /api/v1/layers/{layerId}/features/{featureId}`
- `GET /features/{id}/history`
- WebSocket-канал подписки на слой

## 8. Требования к ошибкам

Сервис должен возвращать предсказуемые ошибки:

- `401` — пользователь не авторизован;
- `403` — у пользователя недостаточно прав;
- `404` — слой или объект не найден;
- `409` — конфликт версии;
- `422` — невалидный payload, geometry, coordinates или `bbox`.

Должны поддерживаться типизированные error-body как минимум для следующих случаев:

- `VERSION_MISMATCH`
- `UNSUPPORTED_GEOMETRY`
- `INVALID_COORDINATES`
- `INVALID_BBOX`
- `VERSION_REQUIRED`

## 9. Требования к хранению данных и БД

- Геометрия должна храниться в PostGIS.
- Таблица `features` должна содержать поля:
  - `id`
  - `layer_id`
  - `geom`
  - `props`
  - `revision` или `version`
  - `updated_at`
- На геометрию должен быть индекс `GiST`.
- Для карты должна использоваться bbox-фильтрация как базовый способ выборки.

## 10. Нефункциональные требования

### 10.1. Развёртывание и запуск

- Проект должен подниматься через `Docker Compose`.
- Проект должен запускаться воспроизводимо одной командой.

### 10.2. CI/CD

- Должен существовать CI pipeline с:
  - форматированием;
  - линтингом;
  - тестами;
  - сборкой.
- Должен существовать smoke-test, например через `/health`.

### 10.3. Архитектура

- Backend должен быть разделён на слои:
  - `api`
  - `services`
  - `repositories`
  - `models`
  - `schemas`
- Frontend должен быть разделён на подсистемы:
  - `features`
  - `shared`
  - `map`
  - `api`
  - `auth`

### 10.4. Наблюдаемость

- Логи должны поддерживать structured logging.
- Должна поддерживаться корреляция запросов через `request id`.

## 11. Критерии приёмки MVP

MVP считается готовым, если одновременно выполняются следующие условия:

- пользователь логинится и получает токен;
- карта отображает GeoJSON-слой из PostGIS;
- `Editor` может создать, изменить и удалить объект;
- второй клиент видит изменения по WebSocket за `<= 1–2 сек`;
- конфликт версий возвращает `409` и корректно обрабатывается на клиенте;
- pipeline зелёный, проект воспроизводимо запускается;
- существует документация запуска, API и ограничений MVP.

## 12. Приоритет демонстрационного сценария

Ключевой сценарий демонстрации должен быть таким:

1. Два пользователя открывают один и тот же слой.
2. Первый пользователь редактирует объект и успешно сохраняет его.
3. Второй пользователь почти сразу видит это изменение по WebSocket.
4. Если второй пользователь пытается сохранить устаревшую версию объекта, он получает `409`.
5. Клиент второго пользователя перезагружает актуальную feature и предлагает повторить редактирование.
