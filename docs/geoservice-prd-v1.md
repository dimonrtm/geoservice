# PRD v1.0 Геосервис совместного редактирования

## 1. Обзор продукта

Геосервис предназначен для совместного редактирования геоданных в браузере.
MVP строится на стеке `Vue + MapLibre` для frontend и `FastAPI + PostGIS` для backend.

Цель MVP: дать двум и более пользователям возможность просматривать один и тот же слой, редактировать объекты и видеть изменения друг друга почти в реальном времени, без молчаливой потери данных.

## 2. Цель спринта / MVP

К концу MVP пользователь должен уметь:

- войти в систему;
- открыть слой на 2D-карте;
- создать, изменить и удалить геообъект;
- получить realtime-обновление изменений другого пользователя;
- корректно обработать конфликт редактирования через версионирование.

## 3. Scope MVP

### In Scope

- JWT-авторизация;
- роли `Viewer` и `Editor`;
- 2D-карта на `MapLibre`;
- хранение геоданных в `PostGIS`;
- GeoJSON как основной формат обмена;
- CRUD для feature;
- выборка объектов по `bbox`;
- WebSocket-обновления по слою;
- optimistic concurrency через `version/revision`;
- базовый audit/history;
- базовый CI/CD и воспроизводимый запуск.

### Out of Scope

- `CRDT/OT`;
- offline-first режим;
- сложные блокировки;
- ACL на уровне объекта и поля;
- сложная топологическая валидация;
- multi-object транзакции;
- полнофункциональная 3D-платформа.

## 4. Пользователи и роли

### Viewer

- может просматривать список слоёв;
- может загружать объекты слоя;
- может видеть обновления в realtime;
- не может создавать, изменять и удалять объекты.

### Editor

- может делать всё, что умеет `Viewer`;
- может создавать, изменять и удалять объекты;
- получает ошибки конфликта версий при попытке сохранить устаревшую редакцию.

## 5. Основные сущности

- `User`
- `Project`
- `Layer`
- `Feature`
- `FeatureEdit` (минимальный журнал изменений)

## 6. Предположения и ограничения

- основной формат геоданных — `GeoJSON`;
- `SRID = 4326`;
- порядок координат — `[lon, lat]`;
- у слоя фиксированный `geometryType`;
- `Feature.geometry.type` должен совпадать с `Layer.geometryType`;
- команда реализации в MVP может быть минимальной, поэтому выбираются простые и надёжные решения вместо дорогих в реализации.

## 7. User Stories

### US-1. Авторизация

Как пользователь,
я хочу войти в систему и получить токен,
чтобы иметь доступ к защищённым возможностям сервиса.

### US-2. Просмотр слоя

Как пользователь,
я хочу открыть слой на карте и видеть объекты в пределах текущего viewport,
чтобы работать с геоданными в нужной области.

### US-3. Создание объекта

Как `Editor`,
я хочу создать новый геообъект на карте,
чтобы добавить новые данные в слой.

### US-4. Редактирование объекта

Как `Editor`,
я хочу изменить геометрию или атрибуты объекта,
чтобы актуализировать данные слоя.

### US-5. Удаление объекта

Как `Editor`,
я хочу удалить объект,
чтобы убрать неактуальные данные.

### US-6. Realtime-обновления

Как пользователь,
я хочу почти сразу видеть изменения, сделанные другим пользователем,
чтобы работать с актуальным состоянием слоя.

### US-7. Обработка конфликта

Как `Editor`,
я хочу получить понятную ошибку, если сохраняю устаревшую версию объекта,
чтобы не потерять чужие изменения и продолжить работу с актуальными данными.

### US-8. Просмотр истории

Как пользователь или администратор продукта,
я хочу видеть минимальную историю изменений объекта,
чтобы понимать, кто и когда его менял.

## 8. Функциональные требования

### 8.1. Авторизация и безопасность

- Система должна выдавать JWT-токен после успешного входа.
- Все защищённые эндпойнты должны требовать авторизацию.
- Неавторизованный пользователь должен получать `401`.
- Пользователь с ролью `Viewer` на write-endpoints должен получать `403`.

### 8.2. Карта и загрузка данных

- Карта должна отображаться в 2D.
- Объекты должны загружаться по `bbox`.
- Запросы должны поддерживать debounce.
- Клиент должен уметь отменять устаревшие запросы.

### 8.3. CRUD объектов

- `Editor` должен иметь возможность:
  - создать объект;
  - изменить объект;
  - удалить объект.
- Сервис должен возвращать актуальную версию объекта после успешной операции.
- Удаление и изменение должны учитывать `version/revision`.

### 8.4. Realtime

- Клиент должен иметь возможность подписаться на события слоя по WebSocket.
- Сервер должен отправлять:
  - `feature_created`
  - `feature_updated`
  - `feature_deleted`
- Изменения должны доходить второму клиенту за `<= 1–2 сек`.

### 8.5. Конфликты редактирования

- Изменение допускается только при совпадении версии из запроса с версией в БД.
- При несовпадении версий сервер должен вернуть `409`.
- Клиент после `409` должен:
  - показать уведомление;
  - сбросить локальные изменения;
  - перезагрузить объект;
  - разрешить повторное редактирование только после успешного `GET`.

### 8.6. История изменений

- Сервис должен логировать факт изменения feature.
- Должен существовать минимальный способ просмотра истории объекта.

### 8.7. Аналитика MVP

Система должна поддерживать минимум 1–2 пространственные операции:

- подсчёт объектов в `bbox`;
- расчёт площади полигона;
- поиск объектов в радиусе от точки.

## 9. Нефункциональные требования

### 9.1. Производительность

- Изменения по WebSocket должны доставляться за `<= 1–2 сек`.
- Карта должна работать по viewport/bbox-выборке, а не через полную выгрузку всех данных.

### 9.2. Надёжность

- Конфликт не должен приводить к молчаливой потере данных.
- Realtime-события должны быть идемпотентны на клиенте.

### 9.3. Поддерживаемость

- Backend должен быть разделён на `api / services / repositories / models / schemas`.
- Frontend должен быть разделён на `features / shared / map / api / auth`.

### 9.4. Развёртывание

- Проект должен запускаться через `Docker Compose`.
- CI должен включать форматирование, линтинг, тесты и сборку.

### 9.5. Наблюдаемость

- Должны быть structured logs.
- Должен поддерживаться `request id`.

## 10. Требования к данным

- GeoJSON — основной формат обмена.
- `SRID=4326`.
- Координаты:
  - `lon ∈ [-180, 180]`
  - `lat ∈ [-90, 90]`
- `bbox` должен быть валидным:
  - `minLon < maxLon`
  - `minLat < maxLat`
- Допустимые типы геометрий:
  - `Point`
  - `LineString`
  - `Polygon`
  - `MultiPoint`
  - `MultiLineString`
  - `MultiPolygon`

## 11. API-контракт MVP

Минимально необходимы:

- `GET /api/v1/layers`
- `GET /api/v1/layers/{layerId}/features?bbox=...&limit=...`
- `GET /api/v1/layers/{layerId}/features/{featureId}`
- `POST /api/v1/layers/{layerId}/features`
- `PATCH /api/v1/layers/{layerId}/features/{featureId}`
- `DELETE /api/v1/layers/{layerId}/features/{featureId}`
- `GET /features/{id}/history`
- WebSocket-канал подписки на слой

## 12. Ошибки и ожидаемое поведение

Сервис должен предсказуемо возвращать:

- `401 Unauthorized`
- `403 Forbidden`
- `404 Not Found`
- `409 Conflict`
- `422 Unprocessable Entity`

Минимальные типы ошибок:

- `VERSION_MISMATCH`
- `UNSUPPORTED_GEOMETRY`
- `INVALID_COORDINATES`
- `INVALID_BBOX`
- `VERSION_REQUIRED`

## 13. Acceptance Criteria

### AC-1. Авторизация

- Пользователь может войти и получить токен.
- Без токена защищённые эндпойнты недоступны.

### AC-2. Просмотр карты

- Пользователь видит слой на 2D-карте.
- Объекты слоя загружаются из backend в пределах `bbox`.

### AC-3. CRUD

- `Editor` может создать объект.
- `Editor` может изменить объект.
- `Editor` может удалить объект.

### AC-4. Realtime

- Если пользователь A меняет объект, пользователь B видит изменение не позднее чем через `1–2 секунды`.

### AC-5. Конфликт версий

- Если пользователь A сохранил объект с версией `7` и получил версию `8`,
- а пользователь B пытается сохранить ту же feature с версией `7`,
- B получает `409 VERSION_MISMATCH`,
- после чего клиент B перезагружает актуальный объект.

### AC-6. Ролевой доступ

- `Viewer` не может использовать `POST/PATCH/DELETE`.
- `Editor` может использовать `GET/POST/PATCH/DELETE`.

### AC-7. Готовность поставки

- CI зелёный.
- Проект воспроизводимо запускается.
- Есть документация запуска и описание API/WS.

## 14. Definition of Done

Задача считается завершённой, если:

- реализована в коде;
- покрыта достаточной автоматической проверкой;
- не ломает ключевой demo-сценарий;
- задокументирована, если меняет публичный контракт или поведение системы.

## 15. Ключевой demo-сценарий

1. Пользователь A и пользователь B открывают один и тот же слой.
2. Пользователь A изменяет объект и успешно сохраняет его.
3. Пользователь B почти сразу видит изменение по WebSocket.
4. Если пользователь B сохраняет старую версию объекта, он получает `409`.
5. Клиент B перезагружает актуальную feature и предлагает повторить редактирование.

## 16. Риски MVP

- недооценка сложности realtime и конфликтов;
- недостаточное покрытие интеграционными и e2e тестами;
- деградация производительности на больших bbox;
- смешение dev-auth и production-конфигурации;
- рост сложности карты без достаточной изоляции map-логики.
